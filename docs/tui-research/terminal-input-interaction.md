# Terminal Input and Interaction: Exhaustive Research Report

**Audience:** rocket_surgeon core team, TUI intermission design phase.
**Purpose:** Comprehensive reference for every input mechanism, encoding scheme, and interaction pattern available in modern terminals. Covers what works, what breaks, and what to build on.
**Stance:** Opinionated where it matters. The goal is informed design decisions for a multi-GPU debugger TUI built on ratatui/crossterm.

---

## 1. Keyboard Input Fundamentals

### 1.1 How Keystrokes Become Bytes

The pipeline from physical key to application byte is: **keyboard hardware** -> **OS keyboard driver** (produces keycodes + modifier state) -> **terminal emulator** (translates to byte sequences per its configuration) -> **PTY master side** (kernel pseudoterminal layer) -> **PTY slave side** (the fd the application reads from). On the application side, the fd is a plain Unix file descriptor; the application has no direct access to the keyboard. It receives only the byte stream the terminal chose to emit.

This architecture means the terminal emulator is the *sole authority* on what bytes an application sees for any given keystroke. There is no out-of-band channel. The application must *parse* byte sequences to reconstruct the user's intent, and the mapping from intent to bytes is terminal-dependent.

On Windows, the pipeline is different: the console subsystem provides `ReadConsoleInputW`, which delivers structured `INPUT_RECORD` events containing virtual key codes, scan codes, and modifier state. crossterm abstracts this into the same event model it uses on Unix. The Windows path is richer in raw information but narrower in terminal diversity.

### 1.2 The Encoding Problem

When the terminal emulator wants to communicate "the user pressed F5" or "the user pressed Ctrl+Right", it must encode that as a sequence of bytes. The encoding schemes, accumulated over 50 years, are a layered mess:

**Plain ASCII keys (0x20-0x7E):** These are unambiguous. Pressing `a` sends `0x61`. This is the only category that Just Works.

**Control characters (0x00-0x1F):** Generated by holding Ctrl and pressing a letter. The terminal sends the ASCII value of the uppercase letter minus 64. This means Ctrl+A = 0x01, Ctrl+Z = 0x1A. The problem: several of these collide with named control characters. **Ctrl+H = 0x08 = Backspace. Ctrl+I = 0x09 = Tab. Ctrl+J = 0x0A = Line Feed (Enter on some systems). Ctrl+M = 0x0D = Carriage Return (Enter).** Ctrl+[ = 0x1B = Escape. There is no way to distinguish "the user pressed Tab" from "the user pressed Ctrl+I" in legacy encoding -- they produce identical bytes.

**Escape sequences (CSI, SS3):** For keys that have no ASCII representation, terminals send multi-byte escape sequences:
- **CSI (Control Sequence Introducer):** `ESC [` (0x1B 0x5B), followed by parameters and a final byte. Example: `ESC [ A` = Up arrow. `ESC [ 15 ~` = F5.
- **SS3 (Single Shift 3):** `ESC O` (0x1B 0x4F), followed by a single byte. Example: `ESC O P` = F1 (in xterm). SS3 is used for F1-F4 and sometimes for arrow keys in application mode.
- **OSC (Operating System Command):** `ESC ]` followed by a numeric parameter and string data, terminated by `ST` (ESC \) or `BEL` (0x07). Used for clipboard, window title, etc. -- not typically for keyboard input but critical for other interaction.

**The ambiguity problem:** Because all escape sequences start with `ESC` (0x1B), and the Escape *key* also sends 0x1B, applications must use *timing heuristics* to determine whether a bare 0x1B is "the user pressed Escape" or "the start of a multi-byte sequence whose remaining bytes haven't arrived yet." Typical timeout: 50-100ms. This is inherently fragile -- on slow connections (SSH, mosh), the timeout can fire incorrectly.

### 1.3 Function Keys F1-F24: Encoding Chaos

Function key encoding is a disaster zone. The historical layers:

- **VT220 (1983):** Defined F6-F20 as `CSI nn ~` where nn is 17-34 (with gaps: 22, 23, 24 are F11, F12, F13). F1-F5 were *not defined* on the VT220 -- the physical keys above the keyboard were labeled PF1-PF4 and sent SS3 P/Q/R/S.
- **XFree86 xterm (late 1990s):** Used SS3 P/Q/R/S for F1-F4, `CSI 15 ~` for F5, and VT220 codes for F6-F12. Modified function keys (Shift+F1, etc.) added `;modifier` before the final character. The VT100 PF keys and VT220 function keys were mixed into one key map.
- **VTE (GNOME Terminal, etc.):** Followed xterm but with historical differences around whether modified F1-F4 use SS3 or CSI prefix. Older VTE sent SS3 for modified function keys; newer versions match xterm.
- **F13-F24:** Some terminals implement these as Shift+F1 through Shift+F12. Some terminals do not implement them at all. Some use the same codes as unmodified keys with different modifier parameters. The numbering of shifted function keys (F10-F22 vs F13-F24) varies by terminal and terminfo entry.

**Practical consequence:** An application that wants to bind F13 or Shift+F5 must either (a) use the Kitty keyboard protocol, (b) detect the terminal and use terminal-specific tables, or (c) accept that it will break on some terminals.

Thomas Dickey maintains the authoritative [table of function-keys for xterm and other terminal emulators](https://invisible-island.net/xterm/xterm-function-keys.html) -- it has *eight* different keyboard models listed, each with different encodings.

### 1.4 Modifier Key Detection

**Ctrl:** Reliable for letters (maps to 0x01-0x1A), but collides with named control codes (see 1.2). Ctrl+punctuation is inconsistent across terminals. Ctrl+digits are mostly unsupported in legacy encoding. Ctrl+Shift is indistinguishable from Ctrl alone (both clear the same bits).

**Alt/Meta:** Two encoding strategies exist. Most modern terminals send `ESC` followed by the key's normal encoding ("ESC-prefix" mode). Some can be configured to set the 8th bit instead (producing bytes 0x80-0xFF), which conflicts with UTF-8 and is essentially dead. The ESC-prefix approach creates the Escape-key ambiguity described in 1.2.

**Shift:** For printable characters, Shift just changes the character (a -> A). For function/arrow keys with CSI encoding, Shift adds `;2` to the parameter. Shift alone on a printable key is invisible to the application in legacy mode -- it just sees 'A', not 'Shift+a'.

**Super/Hyper:** Not transmitted at all in legacy terminal encoding. These modifiers are intercepted by the window manager / desktop environment and never reach the terminal emulator. Only the Kitty keyboard protocol can report them.

### 1.5 The Kitty Keyboard Protocol

The Kitty keyboard protocol, designed by Kovid Goyal (author of the kitty terminal) and published in 2021, is the most comprehensive fix for terminal keyboard encoding. It extends Paul "LeoNerd" Evans's `fixterms` CSI-u encoding with:

**Key features:**
- **Unambiguous encoding:** Every key event is encoded as `CSI unicode-codepoint ; modifiers u`. No collisions between Tab and Ctrl+I, between Escape and the start of a sequence, between Enter and Ctrl+M.
- **Modifier flags:** All modifier combinations are reported as a bitmask in the second parameter: Shift=1, Alt=2, Ctrl=4, Super=8, Hyper=16, Meta=32, CapsLock=64, NumLock=128. This gives full access to Super and Hyper, which legacy encoding cannot report.
- **Key press/repeat/release events:** The protocol can report key-down, key-repeat, and key-up as distinct events. Legacy terminals only report key-down (or rather, the character produced). This enables held-key detection, which is important for game-like UIs and continuous-scroll behaviors.
- **Progressive enhancement flags:** Applications opt in to specific capabilities via a flags bitmask, from basic disambiguation (flag 1) up to full press/repeat/release reporting (flag 8). This is opt-in and backwards-compatible.
- **Push/pop stack:** Applications push their desired flags on entry and pop on exit, so nested applications (e.g., vim inside tmux) restore the outer application's state automatically. The stack mechanism is critical for composability.
- **Associated text:** Flag 16 allows the terminal to report the text that a key event would produce alongside the key identity, which is important for dead keys and compose sequences.

**Protocol detection:** Applications query support by sending `CSI ? u` and checking the response. If the terminal does not understand the query, it produces no response (or garbage), and the application falls back to legacy mode.

**Terminal support (as of early 2026):** kitty, WezTerm, foot, ghostty, iTerm2, Alacritty, Rio, Contour, and Windows Terminal (Preview 1.25+). Notably absent: macOS Terminal.app, older GNOME Terminal/VTE builds, mosh. tmux has partial support (passthrough) but does not fully implement the protocol itself.

**This is the protocol rocket_surgeon should target.** The fallback to legacy encoding must exist, but every binding decision should be tested against Kitty protocol semantics first.

### 1.6 modifyOtherKeys and fixterms: The Intermediate Solutions

Before the Kitty protocol, there were two intermediate approaches:

**xterm modifyOtherKeys (2006, Thomas Dickey, xterm patch #214):** Mode 1 sends modified keys that would otherwise be ambiguous as `CSI 27 ; modifier ; keycode ~`. Mode 2 sends *all* keys (including unmodified ones) in this format. Enabled via `CSI > 4 ; 1 m` or `CSI > 4 ; 2 m`.

**fixterms / CSI-u (2008, Paul "LeoNerd" Evans):** Proposed a cleaner encoding: `CSI keycode ; modifier u` with a `u` terminator. Dickey added `formatOtherKeys` to xterm (patch #235) to support Evans's format alongside his own. Evans published the fixterms specification and wrote libtermkey, which Neovim adopted.

**The relationship:** The Kitty keyboard protocol is a superset of fixterms/CSI-u. It uses the same `CSI keycode ; modifier u` base format but adds the progressive enhancement flags, push/pop stack, key release events, and associated text. An application that speaks Kitty protocol automatically handles fixterms-only terminals.

### 1.7 Bracketed Paste Mode

Enabled by `CSI ? 2004 h`, disabled by `CSI ? 2004 l`. When enabled, pasted text is bracketed: `CSI 200 ~` ... pasted content ... `CSI 201 ~`. This lets the application distinguish typed input from pasted input -- critical for security (prevents paste-jacking where pasted text contains newlines that execute commands) and for UX (paste into a search field should not trigger per-character callbacks).

crossterm surfaces this as `Event::Paste(String)` when the `bracketed-paste` feature is enabled. **rocket_surgeon must enable this.** A debugger that interprets pasted text as commands would be a usability disaster.

### 1.8 Key Chords and Sequences

Terminal applications use two strategies for complex key bindings:

**Chords:** Multiple modifiers held simultaneously (Ctrl+Shift+P). Limited by encoding -- legacy terminals cannot distinguish many chord combinations. Kitty protocol makes most chords available.

**Sequences:** Keys pressed one after another (e.g., `g g` in vim for "go to top", or `C-x C-s` in Emacs for save). The application must maintain a state machine that accumulates partial sequences with a timeout. Sequences have no terminal-level support -- they are purely application-level constructs. They do not conflict with terminal emulator bindings because each individual keystroke in the sequence is a normal key event.

**Terminal emulator conflicts:** Many terminal emulators reserve key bindings. Ctrl+Shift+C/V for copy/paste is near-universal. Ctrl+Tab for tab switching. Ctrl+- and Ctrl+= for font size. These bindings *never reach the application* -- they are consumed by the terminal emulator. Some terminals (kitty, WezTerm) allow remapping or disabling these bindings; most do not. **Any key binding design must account for terminal-consumed chords.**

## 2. Mouse Input

### 2.1 Mouse Tracking Modes

Mouse tracking is activated by sending DEC private mode sequences. The modes are cumulative -- each higher mode is a superset:

**X10 mouse reporting (mode 9):** Reports only button press events. No release, no motion. Enabled by `CSI ? 9 h`. Nearly obsolete.

**Normal tracking / X11 mouse (mode 1000):** Reports button press and release. No motion between press and release. This is the baseline for most TUI mouse support.

**Button-event tracking (mode 1002):** Reports motion events, but only while a button is held down (drag). Useful for drag-to-select, resizing panes, etc.

**Any-event tracking (mode 1003):** Reports all mouse motion, even with no buttons pressed. Enables hover effects. Significantly increases event volume -- the application receives a mouse event for every cell the cursor crosses.

**Scroll wheel:** Encoded as button 4 (scroll up) and button 5 (scroll down) with 64 added to the button code in the event byte. No release events for scroll. Scroll events are available in all tracking modes from mode 1000 onward.

### 2.2 Mouse Encoding Formats

The raw mouse event data has its own encoding layer, and this is where the real trouble lives:

**X10/Normal encoding (the default):** `CSI M Cb Cx Cy` where Cb is button+32, Cx is column+32, Cy is row+32. The +32 ensures all bytes are printable ASCII. **Fatal flaw:** coordinates are limited to 223 (255-32), so terminals wider or taller than 223 cells cannot report accurate positions. This format also cannot report button release identities (all releases look the same).

**UTF-8 encoding (mode 1005):** Extends coordinates to 2047 by encoding them as UTF-8 characters. Partially solves the size limit but introduces parsing complexity and still cannot distinguish which button was released. Now considered deprecated.

**SGR encoding (mode 1006):** The modern standard. Format: `CSI < button ; x ; y M` for press, `CSI < button ; x ; y m` for release. Coordinates are decimal integers with no practical upper limit. Button identity is preserved on release. Modifier keys (Shift, Alt, Ctrl) are encoded in the button value. **This is the only encoding rocket_surgeon should use.** Enable it with `CSI ? 1006 h`.

**SGR-Pixels (mode 1016):** Like SGR but reports pixel coordinates instead of cell coordinates. Supported by xterm and a few others. Useful for pixel-precise interaction (e.g., clicking within a rendered heatmap). Support is narrow -- most terminals do not implement it. crossterm does not currently expose pixel coordinates.

### 2.3 Mouse in tmux/screen

Terminal multiplexers intercept and re-emit mouse events. tmux's `set-option -g mouse on` enables mouse support. tmux translates between the outer terminal's mouse encoding and the inner application's requested mode. This generally works but introduces edge cases:
- Coordinates may be offset if the tmux pane does not start at (1,1).
- Mouse mode changes by the inner application may not propagate correctly through older tmux versions.
- SGR encoding works through tmux 2.1+.
- Scroll events in alternate screen mode may be translated to arrow key presses depending on tmux configuration.

### 2.4 Mouse Pixel Position Reporting

Beyond mode 1016 (SGR-Pixels), applications can query the terminal's pixel dimensions:
- `CSI 14 t` reports the text area size in pixels as `CSI 4 ; height ; width t`.
- `CSI 18 t` reports the text area size in characters as `CSI 8 ; rows ; cols t`.
- Dividing pixel dimensions by character dimensions gives cell size in pixels, which is needed for proper sixel/image rendering and for mapping pixel mouse coordinates to cell positions.

Not all terminals support these queries. Windows Terminal notably does not report pixel sizes via CSI 14t.

## 3. Clipboard Integration

### 3.1 OSC 52

OSC 52 is the terminal escape sequence for clipboard access. Format: `ESC ] 52 ; selection ; base64-data BEL` (or `ST` as terminator).

**Selection targets:** `c` = CLIPBOARD, `p` = PRIMARY (X11), `s` = SELECT, `0`-`7` = cut buffers. Most applications use `c`.

**Write (copy):** Send `ESC ] 52 ; c ; <base64-encoded-text> BEL`. Widely supported: kitty, WezTerm, iTerm2, Alacritty, foot, Windows Terminal, tmux (3.3+), and others.

**Read (paste):** Send `ESC ] 52 ; c ; ? BEL`. The terminal responds with the clipboard contents base64-encoded. **This is a security risk** -- a malicious script could silently read the clipboard. Most terminals either block read entirely, prompt the user, or require explicit configuration to allow it. kitty blocks it by default. iTerm2 prompts. Many terminals simply do not implement read.

**Size limit:** Many terminals cap OSC 52 payload at 100,000 bytes (approximately 74,994 bytes of decoded content).

**tmux integration:** tmux 3.3+ supports `allow-passthrough` for OSC 52. The `set-clipboard` option controls whether tmux intercepts or forwards OSC 52. When set to `external`, tmux passes the sequence to the outer terminal. When set to `on`, tmux both sets its own buffer and passes through. This is the primary mechanism for clipboard access from applications inside tmux over SSH.

**SSH forwarding:** OSC 52 works over SSH because it is just bytes in the terminal stream -- the SSH client's terminal emulator processes them. This is the *only* clipboard mechanism that works transparently over SSH without X11 forwarding or external tools.

### 3.2 X11 Selections: PRIMARY vs CLIPBOARD

On X11 systems, there are two independent clipboard buffers:
- **PRIMARY:** Set by selecting text (no explicit copy action). Pasted with middle-click.
- **CLIPBOARD:** Set by explicit copy (Ctrl+C). Pasted by explicit paste (Ctrl+V).

Terminal emulators typically put selected text into PRIMARY. The system clipboard (CLIPBOARD) is separate. OSC 52 targets can address either. Wayland compositors use a different protocol (wl-copy/wl-paste) but the semantics are similar.

## 4. Focus and Window Events

### 4.1 Focus In/Out Reporting

Enabled by `CSI ? 1004 h`. When the terminal window gains focus, it sends `CSI I`. When it loses focus, it sends `CSI O`. crossterm surfaces these as `Event::FocusGained` and `Event::FocusLost` after `EnableFocusChange` is executed. **rocket_surgeon should use this** to pause expensive rendering when the window is not visible and to provide visual cues about active state.

### 4.2 Terminal Resize

**Unix:** The kernel sends `SIGWINCH` to the foreground process group when the terminal size changes. The application reads the new size via `ioctl(fd, TIOCGWINSZ, &winsize)`. crossterm handles this and surfaces it as `Event::Resize(cols, rows)`.

**Windows:** Resize events arrive as `WINDOW_BUFFER_SIZE_EVENT` in the console input stream. crossterm translates to the same `Event::Resize`.

**In-band resize notification (DEC private mode 2048):** A newer mechanism where the terminal sends `CSI 48 ; rows ; cols t` when the size changes. This avoids the SIGWINCH signal path entirely. Useful for applications that want resize notification through the same event stream as keyboard/mouse input. Support is currently limited (contour, a few others).

### 4.3 Cursor Position Reporting

`CSI 6 n` (Device Status Report) causes the terminal to respond with `CSI row ; col R`. This is used to detect the current cursor position, which is useful for applications that need to know where the terminal has placed the cursor (e.g., for inline completion overlays). The response arrives in the input stream and can be confused with the F3 key in some terminals (both produce sequences ending in `R`), so applications must be careful about when they issue this query.

### 4.4 Window Title

Set with `OSC 2 ; title BEL`. Query with `CSI 21 t` (some terminals respond with `OSC l title ST`). rocket_surgeon should set the window title to indicate the current state: model name, current layer, paused/running status.

## 5. Interaction Patterns for Complex TUIs

### 5.1 Modal Editing (Vim-style)

The dominant pattern for complex TUI interaction. The application has distinct *modes* where the same keys do different things:

- **Normal mode:** Single keys are commands (h/j/k/l for navigation, s for step, c for continue, etc.). This maximizes key economy.
- **Insert/input mode:** Keys produce text (for search bars, command input, expression evaluation).
- **Visual/selection mode:** Keys extend or modify a selection.

GDB's TUI uses a variation: `C-x C-a` toggles TUI mode on/off. In TUI mode, arrow keys scroll the source window; in command mode, arrow keys navigate command history. GDB also has a "SingleKey" mode where single letters map to debugger commands (c=continue, n=next, s=step, r=run, etc.).

**The mode indicator is critical.** Users must always know which mode they are in. A status bar element or cursor shape change (via `CSI 6 SP q` for bar cursor, `CSI 2 SP q` for block cursor) communicates mode visually.

### 5.2 Emacs-style Chord Sequences

The alternative to modal editing: multi-key chord prefixes. `C-x` is a prefix -- `C-x C-s` saves, `C-x C-f` opens. The application maintains a key accumulator and a prefix tree of bindings. Advantages: no mode confusion. Disadvantages: more keystrokes per action, harder to discover, and Ctrl+key limitations in terminals make some Emacs bindings impossible (e.g., `C-;` is not sendable in legacy encoding).

### 5.3 Command Palettes

Fuzzy-finder command palettes (popularized by VS Code, implemented in terminal by fzf, telescope.nvim, etc.) are the modern discovery mechanism. The pattern: a hotkey (often `:` or `Ctrl+P` or `/`) opens an input field, the user types a partial match, a ranked list of commands/items updates live, and Enter selects. This requires:
- Fast text input capture (input mode)
- Efficient fuzzy matching (Rust has excellent crates: `nucleo`, `fuzzy-matcher`)
- Rapid re-rendering of the filtered list (ratatui's immediate-mode rendering is ideal here)

### 5.4 Panel Navigation and Focus Management

Multi-pane TUIs (which rocket_surgeon will be) need focus management:
- Tab or a dedicated key (e.g., Ctrl+W followed by arrow, vim-style) switches focus between panes.
- The focused pane has a visual indicator (highlighted border, different border color).
- Some keys are global (quit, help), some are pane-local (scroll, select).
- ratatui does not provide built-in focus management -- the `ratatui-interact` crate or custom state tracking is needed.

### 5.5 Scrolling

Scroll wheel events (via mouse tracking) naturally map to scrolling the focused pane. Arrow keys or j/k in normal mode provide keyboard scrolling. Page Up/Page Down are available via CSI sequences. For large content (tensor views with thousands of elements), virtual scrolling (rendering only visible rows) is essential -- ratatui's `Table` and `List` widgets support this natively with offset tracking.

### 5.6 Text Input Within TUI

Search bars, expression evaluators, and command bars require inline text editing. This means handling:
- Printable character insertion at cursor
- Backspace/Delete
- Left/Right cursor movement (Home/End for line boundaries)
- History navigation (Up/Down in a command bar)
- Ctrl+A (beginning of line), Ctrl+E (end of line), Ctrl+W (delete word)
- Paste (via bracketed paste)

The `tui-input` and `tui-textarea` crates provide this for ratatui, but a debugger likely needs a custom implementation for expression editing with syntax highlighting.

## 6. Advanced Interaction

### 6.1 Drag and Drop

True drag-and-drop (file drag from a file manager into the terminal) is not a terminal protocol feature. Some terminal emulators (iTerm2, kitty) have proprietary extensions. iTerm2 can receive dropped files as a proprietary escape sequence. For practical purposes, **drag-and-drop is not available in terminals.** Mouse drag *within* the TUI (button-event tracking, mode 1002) is available and can implement pane resizing, selection, etc.

### 6.2 IME (Input Method Editor) for CJK

IME support is handled by the terminal emulator, not the application. The terminal emulator interfaces with the OS IME framework (IBus/Fcitx5 on Linux, macOS input sources, Windows IME) and sends the *composed* characters to the application. From the application's perspective, IME input arrives as normal Unicode characters -- the composition UI is rendered by the terminal emulator in an overlay.

**Caveat:** Some terminal emulators have poor IME support. The cursor position for the IME composition window is often wrong in alternate-screen-buffer mode because the terminal does not know where the application's logical cursor is. Applications can mitigate this by setting the terminal cursor position to where text input is expected, even if they are otherwise managing their own cursor.

### 6.3 Right-to-Left Text

Terminal handling of RTL (Arabic, Hebrew) text is inconsistent. The Unicode Bidirectional Algorithm is complex. Most terminal emulators either do not implement it or implement it partially. For rocket_surgeon this is a non-issue for the MVP -- debugger UIs are inherently LTR.

### 6.4 Undo/Redo

Undo/redo is purely application-level state management. The terminal provides no undo mechanism. Vim's undo tree is the gold standard: every edit creates a node in a branching tree, not a linear stack. For a debugger, undo applies to different things:
- Undo navigation (go back to previously viewed layer/tensor)
- Undo surgical modification (revert an activation edit)
- Undo breakpoint changes

Each requires its own state history. A command pattern (storing inverse operations) or snapshot pattern (storing state checkpoints) are the standard approaches.

### 6.5 Touch Input

Mobile terminals (Termux on Android, a-Shell on iOS) translate touch events to mouse events. Tap = click, swipe = scroll. The touch-to-mouse translation is done by the terminal emulator. From the application's perspective, these are normal mouse events. The main design implication: touch targets must be large enough (minimum 44x44 pixels, or roughly 5-6 character cells) and hover-dependent interactions must have tap alternatives.

## 7. crossterm Specifics

### 7.1 What crossterm Abstracts

crossterm provides a unified event model across platforms:
- `Event::Key(KeyEvent)` with `KeyCode`, `KeyModifiers`, and `KeyEventKind` (Press/Repeat/Release)
- `Event::Mouse(MouseEvent)` with button, modifiers, column, row, and `MouseEventKind` (Down/Up/Drag/Moved/ScrollDown/ScrollUp)
- `Event::Resize(cols, rows)`
- `Event::FocusGained` / `Event::FocusLost`
- `Event::Paste(String)` (when bracketed-paste feature is enabled)

### 7.2 Kitty Protocol in crossterm

crossterm supports the Kitty keyboard protocol via `PushKeyboardEnhancementFlags` and `PopKeyboardEnhancementFlags`. The flags are:

- `DISAMBIGUATE_ESCAPE_CODES` (flag 1): Eliminates Ctrl+I vs Tab, Escape vs ESC-prefix, etc.
- `REPORT_EVENT_TYPES` (flag 2): Enables key repeat and release events.
- `REPORT_ALTERNATE_KEYS` (flag 4): Reports the key that would have been produced without modifiers.
- `REPORT_ALL_KEYS_AS_ESCAPE_CODES` (flag 8): Even unmodified printable keys get CSI-u encoding.
- `REPORT_ASSOCIATED_TEXT` (flag 16): Includes the text a key event would produce.

crossterm surfaces enhanced keyboard data through the same `KeyEvent` struct, with `KeyEventKind::Repeat` and `KeyEventKind::Release` variants only available when the appropriate enhancement flags are active.

### 7.3 Mouse Support in crossterm

`EnableMouseCapture` activates any-event tracking (mode 1003) with SGR encoding (mode 1006). crossterm handles the protocol negotiation. Mouse events include cell coordinates but **not** pixel coordinates -- SGR-Pixels (mode 1016) is not currently supported by crossterm.

### 7.4 Platform Differences

**Unix (Linux, macOS):** crossterm reads from the TTY file descriptor. Raw mode is set via termios. Behavior is consistent across Unix-like systems, but the *terminal emulator* determines which escape sequences are actually sent.

**Windows:** crossterm uses `ReadConsoleInputW` for input. Raw mode implementation differs from Unix. Windows Terminal (2019+) supports ANSI/VT sequences including mouse and Kitty protocol, but legacy `cmd.exe` and ConHost have limited support.

**macOS Terminal.app:** Does not support the Kitty keyboard protocol. Does not support several xterm extensions. Alt/Option key behavior depends on the "Use Option as Meta key" setting -- when disabled, Option produces Unicode characters (e.g., Option+S = ß on US keyboard). This is a constant source of bug reports. **rocket_surgeon should document that macOS Terminal.app provides a degraded experience** and recommend kitty, WezTerm, iTerm2, or Ghostty on macOS.

### 7.5 Known Limitations

- **No pixel mouse coordinates.** crossterm does not expose SGR-Pixels mode.
- **Keyboard enhancement detection is not foolproof.** Some terminals claim support but have incomplete implementations. crossterm's `supports_keyboard_enhancement()` queries the terminal but edge cases exist.
- **Event parsing ambiguities.** The ESC-timeout problem for legacy terminals is real. crossterm uses a default timeout of ~300ms for incomplete escape sequences.
- **No terminal identification.** crossterm does not query the terminal type (DA1/DA2 sequences). Applications that need terminal-specific behavior must do their own detection.

## 8. Design Implications for rocket_surgeon

### 8.1 Key Binding Strategy

GDB TUI's SingleKey mode is the closest prior art: single letters for frequent debugger commands in normal mode. Recommended bindings for a debugger:

| Action | Key (Normal Mode) | Rationale |
|--------|-------------------|-----------|
| Step into | `s` | GDB convention |
| Step over | `n` | GDB convention ("next") |
| Step out | `f` | GDB convention ("finish") |
| Continue | `c` | GDB convention |
| Reverse step | `S` (Shift+s) | Uppercase = reverse |
| Reverse continue | `C` (Shift+c) | Uppercase = reverse |
| Set breakpoint | `b` | GDB convention |
| Inspect expression | `e` | Opens input mode |
| Command palette | `:` or `/` | Vim convention |
| Focus next pane | `Tab` | Standard |
| Focus prev pane | `Shift+Tab` | Standard, requires Kitty protocol for reliable detection |
| Quit | `q` | Standard |
| Help | `?` | Standard |

**The tension between text input and shortcuts:** Use modal editing. Normal mode keys are single-letter commands. Pressing `:`, `/`, or `e` enters input mode with a focused text field. Escape returns to normal mode. The mode indicator must be always visible. This is the only workable approach for a complex debugger TUI with both frequent commands and expression/search input.

### 8.2 Mouse Interaction for Tensor Heatmaps

With SGR mouse tracking (mode 1006 + mode 1003), rocket_surgeon can:
- **Hover over heatmap cells** to show tensor values in a tooltip/status bar
- **Click to select** individual neurons/positions for inspection
- **Click-drag** to select tensor regions for comparison or surgical modification
- **Scroll** to navigate through layers or large tensor dimensions
- **Click on architecture tree nodes** to expand/collapse or navigate to that component

Without pixel-level mouse (mode 1016), hover precision is limited to cell granularity. For heatmaps rendered as Unicode block characters (where one cell might represent multiple tensor values), the application must map cell coordinates to tensor indices using the known rendering layout.

### 8.3 Protocol Negotiation on Startup

rocket_surgeon's terminal initialization should:
1. Enter alternate screen buffer (`CSI ? 1049 h`)
2. Enable raw mode
3. Attempt to push Kitty keyboard enhancement flags (DISAMBIGUATE + REPORT_EVENT_TYPES)
4. If Kitty protocol is not supported, fall back to legacy mode with known limitations
5. Enable mouse capture (SGR encoding, any-event tracking)
6. Enable focus reporting
7. Enable bracketed paste
8. Query terminal size (TIOCGWINSZ + optionally CSI 14t for pixel size)
9. Set window title

On exit, all of these must be reversed in the opposite order.

### 8.4 tmux/screen Compatibility

Users running rocket_surgeon inside tmux will encounter:
- Kitty protocol may not pass through (depends on tmux version and `allow-passthrough`)
- Mouse coordinates offset by tmux pane position (crossterm handles this transparently)
- OSC 52 clipboard requires tmux `set-clipboard` configuration
- Focus events require tmux `focus-events on`

rocket_surgeon should detect tmux (check `$TMUX` environment variable) and warn about known limitations or provide a configuration hint.
